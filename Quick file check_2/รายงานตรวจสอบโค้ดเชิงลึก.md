## 3. Runtime Risk Analysis

### 🔴 HIGH — `createDestination()` ไม่มี dedup check ก่อน insert ในเส้นทาง Review Queue
**File:** `src/1_group1_master_db/10_MatchEngine.gs`
**Line:** 1787-1868 (`resolveAndPersistCreate_`), เรียกจาก `reprocessReviewQueue()` (`12_ReviewService.gs:981`) ผ่าน `resolveAndPersist_('CREATE_NEW', ...)`
**Description:** เส้นทางหลัก (`handleAutoMatch_`, line 1296) เรียก `resolveDestination(personId, placeId, geoId)` เช็คซ้ำก่อนเสมอ ✅ แต่ `resolveAndPersistCreate_` (ใช้เฉพาะ reprocess Q_REVIEW) เรียก `createDestination()` ตรงๆ ที่บรรทัด 1867 **โดยไม่เช็ค `resolveDestination()` ก่อน** ทั้งที่ `personId`/`placeId` ในฟังก์ชันนี้อาจเป็น entity ที่ resolve เจอของเดิม (ไม่ใช่สร้างใหม่เสมอไป)
**Why it matters:** ถ้า Q_REVIEW สองแถวถูก resolve มาเจอ Trinity (person+place+geo) เดียวกันในรอบเดียวกัน หรือ `reprocessReviewQueue()` ถูก interrupt กลางคันแล้ว resume จาก checkpoint (มีกลไก resume จริงตามคอมเมนต์ PERF-001) จะเกิด **destination ซ้ำ** — กระทบ transaction integrity และ idempotency ของ M_DESTINATION
**Recommended fix:** เพิ่ม `const destResult = resolveDestination(personId, placeId, geoId); destId = (destResult.status !== 'NOT_FOUND' && destResult.status !== 'INSUFFICIENT') ? destResult.destId : createDestination(...);` — แพทเทิร์นเดียวกับที่ `handleAutoMatch_()` ใช้อยู่แล้ว ไม่กระทบ interface หรือ schema

### 🟡 MEDIUM — `isCompany` flag คำนวณแล้วไม่ถูกบันทึกลง Schema (data loss เงียบ)
**File:** `src/1_group1_master_db/06_PersonService.gs:534-585` (`createPerson`) เทียบกับ `src/O_core_system/01_Config.gs:158-171` (`PERSON_IDX`)
**Description:** `normalizePersonNameFull()` คำนวณ `isCompany` (true/false) ตั้งแต่ 05_NormalizeService แต่ `PERSON_IDX` มี 12 field (index 0-11, จบที่ `PHONETIC_SECONDARY`) — ไม่มี field สำหรับ `isCompany` เลย และ `newRow` ใน `createPerson()` ก็มีแค่ 12 ค่าตรงตาม schema พอดี ไม่มีการเขียนค่านี้ลงชีตที่ไหนเลย
**Why it matters:** เป็น "silent failure" ตรงตามหลักการที่ให้ความสำคัญ — ระบบคำนวณข้อมูลสำเร็จ (ไม่ throw error) แต่ผลลัพธ์หายไปเงียบๆ ทำให้ MatchEngine/รายงานไม่สามารถแยกลูกค้าองค์กร vs บุคคลได้ในอนาคต
**Recommended fix:** เพิ่ม field `IS_COMPANY` ต่อท้าย `PERSON_IDX` (index 12) + เพิ่มคอลัมน์ใน `SCHEMA[SHEET.M_PERSON]` + เพิ่ม `normResult.isCompany` ใน `newRow` ของ `createPerson()`

### 🟢 LOW — Loop กวาดคอลัมน์ใน `03_SetupSheets.gs:514` ใช้ `getRange()` ต่อ iteration
**File:** `src/O_core_system/03_SetupSheets.gs:511-521`
**Description:** for-loop กวาด header row เพื่อล้าง format คอลัมน์ที่ตรงเงื่อนไข เรียก `sheet.getRange(1, col)` ทีละคอลัมน์
**Why it matters:** Severity ต่ำเพราะ bound ด้วยจำนวนคอลัมน์ (มักไม่เกิน ~20-30) และรันเฉพาะตอน Setup (ไม่ใช่ hot path ที่รันบ่อย) — ไม่กระทบ performance จริงในทางปฏิบัติ
**Recommended fix:** ไม่จำเป็นเร่งด่วน ถ้าต้องการ optimize ให้สะสม column indices แล้วใช้ `getRangeList()` ครั้งเดียว

ไม่พบ null reference risk, array out-of-bounds, missing SCHEMA mapping ที่อื่นในจุดที่ตรวจ (cross-check `PERSON_IDX`/`DEST_IDX`/`FACT_IDX` กับ `newRow` arrays ที่เกี่ยวข้อง — จำนวน field ตรงกันหมดยกเว้นกรณี isCompany ข้างต้น)

เอกสารข้อ 8 (ข้อห้ามสำคัญ) เขียนไว้ชัดเจนว่า:
> "**ห้ามใช้ `ShipToAddress` จาก SCG API เป็น anchor ใน Group 2**"
> ตาราง 6.2: ShipToAddress → "เก็บไว้แสดง/อ้างอิง แต่**ไม่ใช้จับคู่**"

แต่โค้ดปัจจุบันใน `17_SearchService.gs` มีส่วนที่เพิ่มเข้ามาใหม่ชื่อ **`[ADD v5.5.022-PATCH1] ShipToAddress Tie-Breaker Policy`** (บรรทัด 79-90, 174-237):
```
* [ADD v5.5.022-PATCH1] ShipToAddress Tie-Breaker Policy:
*   จะใช้ ShipToAddress เป็น tie-breaker โดยเปรียบเทียบกับ address ของแต่ละ Destination
```
คือเมื่อ `ShipToName` แมตช์กับ Master ได้ **หลาย Destination พร้อมกัน** (ชื่อซ้ำ) ระบบจะดึง `ShipToAddress` มาเทียบ (ผ่าน bigram similarity) เพื่อ**เลือกว่าจะใช้ Destination ไหน** — นี่คือการ "ใช้ ShipToAddress ช่วยจับคู่" ในทางปฏิบัติ แม้จะเป็นแค่ตัวช่วยตัดสินใจรอง (secondary tie-breaker) ไม่ใช่ anchor หลัก แต่ก็ขัดกับถ้อยคำที่ doc เขียนห้ามไว้แบบเด็ดขาด

**ควรพิจารณา:** อัปเดตเอกสารให้ระบุข้อยกเว้นนี้ชัดเจน (เช่น "ShipToName ยังเป็น anchor หลักเสมอ ShipToAddress ใช้เป็น tie-breaker รองเมื่อชื่อกำกวมเท่านั้น ไม่ใช่ anchor หลัก") เพื่อไม่ให้ทีมในอนาคตเข้าใจผิดว่า ShipToAddress ถูกห้ามใช้แบบเบ็ดเสร็จ
